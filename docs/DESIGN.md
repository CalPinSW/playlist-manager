## Problem Statement

The user has a personal music ritual: every Friday, browse rateyourmusic.com/new-music,
manually pick the week's most interesting new albums, build a "New Albums DD/MM/YY"
Spotify playlist, and work through it across the following week. Albums they love get
promoted to a "Best Albums DD/MM/YY" playlist via a quick-access button.

The existing web app (Next.js on Vercel) handles the album-first playback experience
well — it splits playlists by album, tracks progress through songs/albums/playlists,
and provides the quick "add to Best Albums" action. But it's web-only. Listening
progress is lost when the tab closes. There's no background tracking on mobile. No
widgets. No offline support.

The full vision: a Personal Listening OS. A single authoritative place where your
taste lives — full listening history, album ratings, genre tracking, semi-automated
weekly discovery, and seamless mobile + desktop sync.

---

## What Makes This Cool

1. **Album-first is genuinely novel.** Every major music tool (Spotify, Last.fm, stats.fm)
   is built around tracks and artists. Nobody has built a consumer tool for the
   album listener. This is the product insight.

2. **The Friday ritual has structure.** Weekly cadence, named playlists with dates,
   a promotion mechanic (New → Best). That structure is the skeleton of a personal
   taste archive. Ratings and genre tags make it searchable over time.

3. **Widgets as ambient awareness.** A home screen widget showing your current album,
   progress, and a quick "star this" action — that's genuinely useful, not gimmicky.

4. **Retroactive history is free.** Spotify's `recently_played` endpoint (server-side)
   can reconstruct where you are in an album even if you listened on your phone with
   no app open. A Trigger.dev scheduled task running every 15 minutes handles this —
   no Vercel Pro required, no app-open trigger needed.

## Constraints

- Single developer, personal use — no scale concerns
- Spotify API only (no RYM API available yet)
- Existing Vercel backend and Spotify OAuth can be reused or extended
- Distribution: personal use, so TestFlight (iOS) or direct APK (Android) is fine —
  no App Store submission required initially
- Budget: Vercel Hobby (free) for the web API; Neon Postgres free tier; Trigger.dev
  free tier for background jobs and scheduling. Trigger.dev runs its own scheduler —
  Vercel's cron tier limits are irrelevant. Any cadence (1 min, 15 min, hourly) is
  available on Trigger.dev's free tier.
- Trigger.dev tasks are running on v4 (upgrade completed separately).

## Premises

1. The core value is album-first listening — progress tracking, full-record playback,
   and quick curation actions (Best Albums). This is what doesn't exist elsewhere.

2. The RYM discovery gap is real but not cleanly solvable today. MusicBrainz can
   provide date-based release lists, but genre/rating quality won't match RYM.
   Semi-automated (MusicBrainz seed → manual curate) is more realistic than fully
   automated.

3. Mobile is the most impactful gap. Progress doesn't survive page refreshes, and
   background tracking requires a native layer or server-side reconstruction.

4. Ratings and genre tracking compound in value over time — they enable future
   discovery ("show me 4-star jazz from 2024"). Worth designing into the data model
   now even if the UI ships later.

5. The vision is phased. Data layer + native app first. Then discovery. Then
   ratings/genre. Then analytics.

## Approaches Considered

### Approach A: PWA Upgrade
Upgrade the existing Next.js app to a PWA with service worker + offline manifest.
Add `recently_played` cron for retroactive progress sync. No native app.

- Effort: S-M (1 weekend)
- Risk: Low
- Pros: Zero new distribution surface, same Vercel deploy, homescreen installable
- Cons: No real background playback control, iOS PWA limitations (no push, limited
  storage), can't control Spotify when app isn't foregrounded
- Reuses: 100% of existing codebase

### Approach B: Data Foundation First, Native App Later
Build `recently_played` cron + Postgres progress table this weekend. Add ratings/genre
schema at the same time. Then build Expo native app once data layer is solid.

- Effort: M now + L later
- Risk: Medium
- Pros: Data useful immediately on all devices, native app built on solid foundation
- Cons: Two codebases temporarily, native app distribution friction

### Approach C: Full Native App (User's Choice)
Build a full Expo + React Native app as the primary client. Web app becomes optional
or deprecated. Expo Router (file-based, Next.js-like) for navigation. Vercel API
remains as backend. SQLite locally for offline. Spotify SDK for real background
events and playback control. Home screen widgets via react-native-widget-extension.

- Effort: L (3-6 weekends)
- Risk: Medium (Spotify RN SDK is community-maintained)
- Pros: Full offline, real background SDK events, widgets, best-in-class mobile UX
- Cons: Bigger investment upfront, distribution via TestFlight/direct APK

### Data model (design now, implement incrementally)

The strategy is to **extend and rename existing Prisma models** rather than add parallel tables.
This avoids dual sources of truth. The existing `playback_state` + `playbackstatealbumrelationship`
will be refactored into a `listening_progress` model.

```prisma
// RENAME: playback_state → listening_progress
// RENAME: playbackstatealbumrelationship → listening_progress_album (extend with new fields)
// RENAME: playbackstateplaylistrelationship → listening_progress_playlist

// New shape in schema.prisma:
model listening_progress {
  id              Int      @id @default(autoincrement())
  user_id         String   @db.VarChar(255)
  album_id        String   @db.VarChar(255)
  playlist_id     String   @db.VarChar(255)
  last_track_index Int     // 0-based, authoritative for UI display
  total_tracks    Int
  listened_at     DateTime
  source          String   @db.VarChar(50)  // 'manual' | 'recently_played'
  user            user     @relation(...)
  album           album    @relation(...)
  playlist        playlist @relation(...)

  @@unique([user_id, album_id, playlist_id])  // upsert key
}

// Sync cursor storage (new)
model sync_log {
  id           Int      @id @default(autoincrement())
  user_id      String   @unique @db.VarChar(255)
  last_played_at DateTime?  // cursor for recently_played pagination
  last_run_at  DateTime @default(now())
  user         user     @relation(...)
}

// Album ratings (new)
model album_rating {
  album_id  String   @id @db.VarChar(255)
  user_id   String   @db.VarChar(255)
  rating    Int      // 1-10 internally; display as 0.5-5.0 half stars (÷2)
  rated_at  DateTime
  notes     String?
  album     album    @relation(...)
  user      user     @relation(...)
  @@unique([album_id, user_id])
}
```

**Genre note:** `albumgenrerelationship` + `genre` already exist in the schema.
Use these for genre tagging — do NOT add a separate `album_genres` table.

**Migration strategy:**
1. Prisma migration to add `listening_progress`, `sync_log`, `album_rating`
2. Data migration: copy existing `playback_state` rows into `listening_progress`
3. Update all web app code reading from `playback_state` to read from `listening_progress`
4. Drop `playback_state`, `playbackstatealbumrelationship`, `playbackstateplaylistrelationship`
   once the web app is updated (do in a follow-up migration after verifying correctness)

### Build order (phased weekends)

**Weekend 1 — Data layer**

Step 0a — Clean up dual test runners (15 minutes):
- Remove from `devDependencies`: `babel-jest`, `jest-environment-jsdom`, `@babel/core`,
  `identity-obj-proxy`, `jest` (if present)
- Keep: `vitest`, `@vitejs/plugin-react`, `@testing-library/react`, `@testing-library/jest-dom`
- Ensure `vitest.config.js` is the single test runner. All new tests go in Vitest.
- Migrate any existing tests in `tests/` that use Jest-only globals to Vitest equivalents.

Then add the new sync task:

- Postgres schema (above) — add to existing Prisma schema or run raw migrations
- New Trigger.dev task `syncRecentlyPlayedTask`:
  ```
  app/trigger/syncRecentlyPlayed.ts
  ```
  Algorithm:
  0. Use shared constant from `app/utils/playlistFilters.ts`:
     ```ts
     export const NEW_ALBUMS_REGEX = /^New Albums \d{2}\/\d{2}\/\d{2}$/;
     export const ALL_ALBUMS_REGEX = /Albums/; // replaces the .includes('Albums') in refreshSpotifyPlaylists
     ```
     Update `refreshSpotifyPlaylists` handler.ts to use `ALL_ALBUMS_REGEX` instead of
     the inline `.includes('Albums')` string check.
  1. Fetch user's Spotify playlists from DB, filter by `NEW_ALBUMS_REGEX`
  2. Call `GET /me/player/recently-played?limit=50&after={cursor}` where cursor is
     the `played_at` timestamp stored in `sync_log` from the last run
  3. Batch resolve: extract all track IDs from `recently_played` response, then do a
     single Prisma query to get track→album→playlist mappings:
     ```ts
     const trackMappings = await prisma.track.findMany({
       where: { id: { in: trackIds } },
       include: {
         album: {
           include: {
             playlistalbumrelationship: {
               where: { playlist_id: { in: activePlaylistIds } }
             }
           }
         }
       }
     });
     ```
     Zero Spotify API calls for reconciliation — all data is already in the DB from
     the existing `updatePlaylistData` sync. Tracks not in any New Albums playlist
     are silently dropped.
  4. Group by album. Take the highest `track_number` seen as `last_track_index`.
     Deduplicate multiple listens to same track by latest `played_at`. Upsert into
     `listening_progress` only if new `last_track_index` > stored value (no regression).
  5. Store the most recent `played_at` as the new cursor in `sync_log`
  6. Handle Spotify 429s with Trigger.dev's built-in retry (already configured in
     `trigger.config.ts` with exponential backoff)
  7. **Token safety:** wrap the `refreshSpotifyAccessToken` fetch + `prisma.access_token.update`
     in a Prisma transaction. Spotify sometimes rotates refresh tokens on use — if the
     fetch succeeds but the DB write fails, the old refresh token is now invalid and
     the next retry will get a permanent 401. Apply this fix to the existing
     `refreshSpotifyAccessToken.ts` function (it affects all Trigger tasks, not just sync).
- Schedule: `cron: '*/15 * * * *'` (every 15 min) — Trigger.dev free tier supports this
- Also call this task on app open (belt and suspenders) for near-real-time feel
- Limitation: `recently_played` returns at most 50 tracks per call. If >50 tracks play
  between cron runs (15 min), history in that gap is lost. Extremely unlikely in practice.
- Test: listen to 3 tracks on phone with web app closed, open app, verify DB

**Tests to write alongside (Vitest, ~30 min):**

```
tests/app/utils/playlistFilters.test.ts
  - NEW_ALBUMS_REGEX matches "New Albums 04/04/26"
  - NEW_ALBUMS_REGEX rejects "Best Albums 04/04/26", "New Albums 4/4/26", "Albums"

tests/app/trigger/syncRecentlyPlayed.test.ts
  - Empty recently_played response → no DB writes
  - Track in New Albums playlist → upserts correct last_track_index
  - Track NOT in any New Albums playlist → silently skipped, no error
  - Existing progress at index 7, new track at index 5 → does NOT regress (stays 7)
  - Existing progress at index 5, new track at index 7 → advances to 7
  - cursor: first run stores played_at of most recent track
  - cursor: second run uses stored cursor as 'after' param
  - Spotify 429 → throws, Trigger retry handles it

tests/app/api/spotify/utilities/refreshSpotifyAccessToken.test.ts
  - Successful refresh → access_token + refresh_token updated atomically
  - DB write fails → original refresh_token NOT invalidated (transaction rollback)
```

**Weekend 2 — Expo app scaffold**
- IMPORTANT: Initialize as a Development Build from day 1, not Expo Go.
  Widgets (Weekend 4) require native extensions that cannot be added to Expo Go.
  Use: `npx create-expo-app playlist-manager-native` then `npx expo install expo-dev-client`
  and set up EAS Build before writing any feature code. `eas build --profile development`
  gets you a `.ipa`/`.apk` you install directly.
- Auth screen → Auth0 PKCE flow via `expo-auth-session` with Auth0 as the identity
  provider. Auth0 issues access + refresh tokens; store them in Expo SecureStore.
  The native app sends the Auth0 access token as a Bearer header to all Vercel API
  calls. The existing `withAuth()` middleware calls `auth0.getSession()` — this needs
  to be extended to also validate Bearer tokens from mobile. Auth0 provides a JWKS
  endpoint to verify the token without a network call to Auth0 on every request.
  On every app open, check token expiry and refresh silently using the stored refresh
  token. Do NOT build a custom JWT issuer — `withAuth()` already handles Auth0 tokens
  and extending it is far less code than maintaining a bespoke token layer.
- Home screen: current album + progress (reads from `/api/progress`)
- Album view: track list, mark progress, skip to track
- Call `/api/sync-history` on every app open (after auth)

**Weekend 3 — Core playback features**
- SDK decision gate (do this before writing playback code): evaluate
  `react-native-spotify-remote` against the current Spotify RN SDK. Check the repo's
  last commit date and open issues. If it looks unmaintained (no commits in 6+ months,
  open PRs rotting), fall back to Expo WebBrowser launching the Spotify app + deep
  links for play/pause. Native SDK is better; web fallback is more stable. Pick one
  and commit.
- react-native-spotify-remote (if chosen): play/pause/skip from app
- SQLite local cache via Expo SQLite: progress and ratings survive offline.
  Note: offline means the UI works without connectivity; Spotify playback itself
  requires a Spotify Premium account and an active internet connection — this is a
  Spotify constraint, not something we can work around.
- Quick action: "Add to Best Albums" calls `/api/promote` (which calls Spotify's
  `POST /playlists/{id}/tracks` server-side)
- Offline conflict resolution: last-write-wins using `listened_at` timestamp. Same
  policy for ratings: if a rating is set offline then a different rating is set on
  web before sync, the later `rated_at` wins. No conflict UI needed.
- Background sync: call `/api/sync-history` on app resume (AppState 'active' event)
- Fallback if SDK not chosen: open `spotify:` URI scheme to hand off playback to the
  Spotify app. No in-app playback control — user controls playback in Spotify, app
  tracks progress via `recently_played` sync only.

**Weekend 4 — Ratings + iOS widget**
- Star rating component on album view (writes to `/api/ratings` → `album_ratings`)
- iOS widget ONLY (Android widget is Weekend 5+, see Future Phases):
  Budget the full weekend for this — App Group setup, entitlement configuration, and
  EAS Build signing changes for `react-native-widget-extension` routinely take a full

  Steps: (1) Add App Group entitlement in Xcode, (2) add WidgetKit extension target,

- Genre tag input UI: deferred to Future Phases. Schema is already in place (Weekend 1).

## Design Decisions (from /plan-design-review on 2026-04-04)

Full review: `great-dijkstra-design-review-20260404.md`

### Navigation structure (IA)

```
Tab bar (4 tabs, persistent):
  ⊙ Now      — hero: current album + progress; This Week carousel
  ♫ Albums   — playlists list → playlist detail → album detail
  ★ Ratings  — all rated albums, filterable by star/date/genre
  ⊕ Discover — HIDDEN until Phase 2 (do not show a disabled tab)

Album Detail: push navigation (not modal) from any tab.
```

### "Current album" definition

`ORDER BY listened_at DESC LIMIT 1` on `listening_progress`. Most recently played. Always correct.

### Rating scale (confirmed)

Store as **1-10 integers** internally. Display as **0.5–5.0 half stars** (value ÷ 2).
Matches RYM scale. Changing display later requires no migration.

### Promote button logic

Always visible on album detail. Prominent green when rating ≥ 4 (implicit suggestion).
Muted with no prompt when no rating yet. Changes to "In Best Albums ✓" (disabled) after promotion.

### Playlist sync on app open

On every app open, call **both**:
1. `syncRecentlyPlayed` — progress update
2. `updatePlaylistData` — playlist discovery (catches new Friday playlists immediately)

The Saturday midnight cron is a safety net, not the primary sync path.
Also add pull-to-refresh on the Albums tab to manually trigger both.

### Interaction states

- Use **skeleton loaders** (not spinners) for album art and track list loading.
- **Optimistic updates** for star ratings and promote action — update UI instantly, revert on failure.
- **Offline:** Show SQLite data immediately with a banner ("Offline — showing last synced data"). No spinner.
- **Empty state (Now tab):** "Nothing playing" with open Spotify CTA if no recent progress found.
- **First-time sync:** Full-screen "Getting your history" state with spinner (only on very first open).

### Component library

**NativeWind** — reuses web Tailwind token names (`primary`, `secondary`, `warning`, `text`).
Single mental model across web and native.

### Additional design tokens needed for native (not in web palette)

```ts
surface: '#0f0a1e'                     // screen background
surfaceElevated: '#24143d'             // cards, sheets, tab bar
surfaceTint: 'rgba(132,61,255,0.08)'   // selected rows
border: 'rgba(255,255,255,0.08)'       // dividers, outlines
```

### Typography

Use **SF Pro** (iOS system font — `fontFamily: undefined`). No custom font imports.
Correct for Retina displays, loads instantly, respects Dynamic Type.

### Layout constraints

- Lock to **portrait** in `app.json`: `orientation: 'portrait'`
- Mark **iPad unsupported** in `app.json`: `supportsTablet: false`
- Target: iPhone SE (375px) as minimum test size before shipping.

### Accessibility (minimum viable, do not skip)

```tsx
<Image accessibilityLabel={`${title} by ${artist}`} />
<View accessibilityRole="progressbar" accessibilityValue={{ min: 0, max: 100, now: pct }} />
<TouchableOpacity accessibilityRole="button" accessibilityLabel={`Rate ${n} out of 5 stars`} />
```

iOS widget: set a single `.accessibilityLabel` on the widget view — WidgetKit renders as image.
