## Pass 1 — Information Architecture

**Rating: 3/10**

The plan names screens ("home screen", "album view") but defines no navigation hierarchy, no tab structure, and no entry/exit flows between screens.

### What the IA needs to be

```
Tab bar (persistent, 4 tabs):
┌─────────────────────────────────────────────────┐
│  ⊙ Now    ♫ Albums    ★ Ratings    ⊕ Discover  │
└─────────────────────────────────────────────────┘

Now (home tab):
  → Hero: currently playing album + progress bar
  → This Week: horizontal scroll of all albums in active New Albums playlist
  → Tap album card → Album Detail (push)

Albums tab:
  → Flat list of all New Albums playlists, newest first
  → Tap playlist → Playlist Detail (push)
    → Tap album → Album Detail (push)

Ratings tab:
  → All rated albums, filterable by star, date, genre
  → Tap → Album Detail (push)

Discover tab:
  → Future Phase 2 (MusicBrainz). Can be hidden or show a "coming soon" state.

Album Detail (modal or push from any tab):
  → Hero with blurred art background
  → Track list with "you are here" divider
  → Star rating (inline, 5 stars)
  → "Promote to Best Albums" CTA (prominent, conditional: only shows if not already promoted)
  → Playlist tag chip (which playlist this is from)
```

### Issues found

**IA-1 (major):** No decision on whether Album Detail is a push navigation or a modal. Push is correct — it's a destination, not an action sheet. A modal makes Back feel wrong if you got there from a track list.

**IA-2 (major):** The "Now" tab is only meaningful if something is currently playing or was recently playing. If Spotify hasn't been opened in 3 days, what does this screen show? The plan says "current album + progress" but doesn't define the no-playing state. The empty state mockup (see empty-state.html) proposes: show the most recently in-progress album with a "Resume" intent, and if nothing has been listened to at all, show the zero-state with a prompt to open Spotify.

**IA-3 (moderate):** The design doc says "Call /api/sync-history on every app open" but doesn't say where the sync result surfaces in the UI. The first time a user opens the app after a listening session, the progress data may not be there yet (sync takes a few seconds). The IA should account for this: show stale data with a "syncing" indicator, update in place when the API call returns.

**IA-4 (minor):** No search defined. Future-proof: the Ratings tab filter (by genre, date, rating) may be enough for v1. Flag for Phase 3.

## Pass 2 — Interaction States

**Rating: 2/10**

Zero states specified in the plan. Every screen needs these:

### Home screen states

| State | Trigger | Design |
|-------|---------|--------|
| Loading | App open, sync in flight | Skeleton placeholder for album art + text — NOT a full-screen spinner |
| Playing | Sync complete, album in progress | Hero layout (see home-screen.html mockup) |
| Syncing update | Sync completes while screen is visible | Progress bar animates to new value; "Synced just now" badge |
| Nothing in progress | No recent plays matching New Albums playlist | Empty state (see empty-state.html) with "Nothing playing" + open Spotify CTA |
| First-time user | Auth complete, first sync running | Full-screen "Getting your history" state (see empty-state.html syncing variant) |
| Offline | No network | Toast: "Offline — showing last synced progress". Data from SQLite. NO spinner. |
| Spotify error | Token expired, Spotify 503 | Inline error on the album card. Not a full-screen error. |

### Album detail states

| State | Trigger | Design |
|-------|---------|--------|
| Loading track list | Album detail opened | Skeleton rows, NOT a spinner |
| Fully listened | All tracks checked off | Progress bar at 100%, green; prompt to rate if not yet rated |
| Unstarted | No progress for this album | Progress bar at 0%, empty track list (all dimmed except track 1) |
| Already promoted | Album is already in Best Albums | "Promote" button → "In Best Albums ✓" (disabled, green) |
| Rating tap | User taps a star | Stars fill immediately (optimistic), API call in background |

### Key principle: optimistic updates everywhere

Rating a star, promoting an album — these should feel instant. Update the UI immediately, fire the API call in the background. If the call fails, revert with a toast. Never make the user wait for a server round-trip to see a response to a tap.

### States NOT needed (keep it simple)

- No "loading" state for the star rating (optimistic update handles it)
- No "undo" for promote — Spotify playlists are not destructive operations
- No elaborate error recovery flow — "Retry" button on any failed sync is enough

## Pass 3 — User Journey

**Rating: 3/10**

The Friday ritual is described well in the design doc. The app journey is not. Here are the four flows that need to be designed:

### Flow 1: Install → First use (Weekend 2)

```
Install TestFlight build
→ Open app
→ Auth0 PKCE screen (Safari browser, Auth0 login page)
→ Authorize Spotify connection
→ Return to app
→ "Getting your history" full-screen state (first sync running)
→ [~5-10 seconds]
→ Home screen with progress populated
```

**Gap:** No onboarding after auth. The user lands directly on their progress (or the empty state). That is probably the right call — just validate it works on a real device before assuming the transition is seamless.

### Flow 2: Friday ritual — New playlist setup

```
Browse RYM → pick albums
→ Build "New Albums 04/04/26" playlist in Spotify (manually, as today)
→ Open app (or wait 15 min for cron)
→ New playlist appears in Albums tab and This Week section of Now tab
→ Tap first album → album detail → Track 1 highlighted
→ Open Spotify, start playing
→ App syncs on next cron run (≤15 min)
→ Home screen shows active album with progress
```

**Gap:** There is a 15-minute delay between creating the playlist in Spotify and the app knowing about it. `updatePlaylistData` runs on Saturday at midnight — this is too infrequent for discovering new playlists during the week. The Friday ritual creates a new playlist on Friday and the user would want to see it in the app immediately. **This is a real UX gap.** On every app open, also call `updatePlaylistData` for the current user (not just `syncRecentlyPlayed`). Or trigger manually with a pull-to-refresh.

### Flow 3: Daily listening (the core loop)

```
Thursday commute:
→ Open Spotify on phone
→ Resume "New Albums 04/04/26" playlist, album 3 of 5
→ Listen to tracks 4-7 of current album
→ [Trigger.dev cron runs ≤15 min later]

Open app at lunch:
→ Home screen shows album 3, track 7 in progress
→ Tap "View Album" → see tracks 1-6 greyed out, track 7 as "you are here"
→ Tap star → rate 4/5 stars
→ Done
```

This flow works. The 15-min cron delay is invisible in normal use.

### Flow 4: Best Albums promotion

```
On album detail:
→ Star rating ≥ 4 → "Add to Best Albums?" prompt appears (contextual, not always visible)
→ Tap "Promote ★" button
→ Optimistic UI: button changes to "In Best Albums ✓"
→ /api/promote calls Spotify POST /playlists/{id}/tracks in background
→ Done
```

**Gap:** "Add to Best Albums?" only shows if a Best Albums playlist exists for the current date period. The app needs to handle the case where no Best Albums playlist exists yet (the user might be in week 1 of the app). Options: auto-create the playlist, or show "Create Best Albums playlist" CTA. The simpler path is to let the user create it in Spotify manually the first time.

## Pass 4 — AI Slop Risk

**Rating: 5/10** (better than average for AI-designed apps — the album-first concept is genuinely specific)

"AI slop" in design means: generic patterns that don't match the actual product. Spotify-clone chrome grafted onto a fundamentally different product concept.

### Risks

**SLOP-1 (high risk): 5-star ratings**
Five stars is the Amazon/movie rating system. RYM uses a 10-point scale. Last.fm uses love/ban. Letterboxd uses 0.5-5 with halves. For someone who cares enough to build this app, 5 stars probably isn't precise enough. Consider: 1-5 with half stars (RYM-style). Or: A/B/C/D/F (letter grades). Or just 0-10 integers.

This is a personal taste tool. The rating scale should feel like yours.

**SLOP-2 (moderate): "Add to Best Albums" as the primary curation action**
This is already well-designed and specific to the ritual. Good. Don't genericise it into a "playlist picker" dialog. Keep it a single-tap action tied to the current Best Albums period.

**SLOP-3 (moderate): Bottom tab bar with generic icons**
Four tabs is right. The icons in the mockup (⊙ ♫ ★ ⊕) are placeholders — the final iconography should feel intentional, not SF Symbols defaults. "Discover" tab is greyed out in v1 anyway; consider hiding it entirely until Phase 2 rather than showing a disabled tab.

**SLOP-4 (low risk): Progress bar as a percentage**
"58%" is precise and useful here. Keep it. Album listening as a percentage is exactly the right unit — it's not slop, it's the actual product concept.

**SLOP-5 (low risk): Dark theme default**
Every music app is dark by default. That's correct here too — album art reads better on dark backgrounds. Not slop; it's the right call. But do NOT do the generic "dark mode auto-follows system setting" as the only option. Make dark the opinionated default, with an option to switch.

### Things that are NOT slop (keep them)

- "You are here" divider in the track list — specific to this product
- Playlist tag chip on album detail — connects the album to the ritual
- "Synced 3 min ago" indicator — transparency about the async nature of progress tracking
- Promote button conditional on rating threshold — builds the curation ritual into the UI

## Pass 5 — Design System Alignment

**Rating: 4/10**

The web app has a defined palette and the native app should inherit and extend it, not start from scratch. But "inherit" needs to mean something specific for native.

### Palette mapping

| Token | Web hex | Native usage |
|-------|---------|-------------|
| primary | #843dff | Accent color: progress bars, active tabs, CTAs |
| primary.lighter | #bea6ff | Secondary text tints, progress fill end |
| primary.darker | #6b04fd | Gradient start on hero backgrounds |
| secondary | #78a63c | "Listened" checkmarks, sync indicator dot, promote success state |
| secondary.lighter | #b3d581 | Promote button text on dark backgrounds |
| warning | #de7c38 | Star ratings (warm, amber-adjacent — correct for this use) |
| text.primary | #24143d | Headline text on light backgrounds ONLY |
| text.secondary | #aeacb0 | Subtext, metadata |
| background | #FFFFFF | NOT USED in native dark mode |

**Gap — native needs additional tokens not in the web palette:**

```ts
// Add to design system:
surface: '#0f0a1e',           // screen background (native dark)
surfaceElevated: '#24143d',   // card/sheet backgrounds
surfaceTint: 'rgba(132,61,255,0.08)', // subtle tint for selected rows
border: 'rgba(255,255,255,0.08)',     // dividers, card outlines
```

### Typography (not defined anywhere)

The web app uses the browser's default font stack. Native needs explicit decisions:

```
SF Pro Display (iOS system) — use it, don't import a custom font
  - Headings: SF Pro Display, semibold/bold, -0.3px tracking
  - Body: SF Pro Text, regular/medium
  - Caption: SF Pro Text, 11-12px, #aeacb0

Do NOT import Geist, Inter, or any web font into the native app.
iOS users expect SF Pro. It reads better on Retina displays.
System fonts also load instantly — zero FOIT risk.
```

### Component library decision (not defined in the plan)

The plan doesn't say what component library (if any) the native app uses. Options:

- **NativeWind** (Tailwind for React Native) — aligns with the web app's Tailwind tokens. Easiest migration path if you want to share some logic.
- **Tamagui** — performant, typed tokens, but complex setup.
- **Plain StyleSheet** — maximum control, no dependencies, more code.

**Recommendation:** NativeWind. It lets you reuse the same color token names from `tailwind.config.ts`. `primary`, `secondary`, `warning` translate directly. The web tokens become the native tokens. This is the right move for a solo developer — one mental model for both surfaces.

### What the web app's RotatingBorderBox and carousel components can't be reused

These are web-only. The native app will need equivalent implementations. The carousel becomes a `FlatList` with `horizontal`. The RotatingBorderBox becomes an Animated value with `useNativeDriver`. These are not hard, but they're not copy-paste.

## Pass 6 — Responsive / Accessibility

**Rating: 3/10**

### Accessibility gaps

**A11Y-1 (major):** No VoiceOver/TalkBack labels specified for any interactive element. Minimum required:

```tsx
// Album art:
<Image accessibilityLabel={`${album.title} by ${artist.name}`} />

// Star rating:
<TouchableOpacity accessibilityLabel={`Rate ${album.title} ${n} out of 5 stars`} accessibilityRole="button" />

// Promote button:
<TouchableOpacity accessibilityLabel="Add to Best Albums playlist" accessibilityRole="button" />

// Track row:
<TouchableOpacity accessibilityLabel={`Track ${n}, ${trackName}, ${duration}`} />
```

**A11Y-2 (moderate):** Progress bar needs to be an `accessibilityRole="progressbar"` with `accessibilityValue={{ min: 0, max: 100, now: progressPct }}`.

**A11Y-3 (moderate):** Minimum tap target size on iOS: 44x44pt. The star rating row with 5 stars at 28px each needs to either be spaced to 44pt minimum or wrapped in a larger hit area.

**A11Y-4 (low):** Color cannot be the only indicator of state. The "listened" tracks use 50% opacity, which is fine. The current track uses a different text color (#bea6ff) AND the ♪ icon — that's correct, two signals.

### Widget accessibility

iOS widgets are NOT accessible in the same way as app screens — they're rendered as images by WidgetKit. VoiceOver reads the widget's `accessibilityLabel` on the whole widget, not individual elements. Set a meaningful label:

```swift
// In the widget SwiftUI view:
.accessibilityLabel("Romance by Fontaines D.C., track 7 of 12, 58% complete")
```

### Responsive layout

- **iPhone SE (375px width):** The hero album art + track list layout should work. Test on this size before shipping — it's the smallest active iPhone.
- **iPad:** Not needed for v1. Personal use on iPhone is the target. Mark explicitly as iPhone-only in `app.json` (`"tablet": false` or `"supportsTablet": false`).
- **Dynamic Type:** Native apps should scale with the user's font size setting. Using `StyleSheet` with fixed px values will NOT scale. NativeWind's font utilities are `rem`-relative on native, which respects Dynamic Type.
- **Landscape:** Lock to portrait for v1. Music apps on iPhone are almost always portrait. `orientation: 'portrait'` in `app.json`.

### Widget size consistency

The widget text uses `Font.system(.caption)` for the artist name and track count. At the minimum widget size (2x2 = 170pt wide), this is fine. At larger sizes, scale up: `Font.system(.body)` for album title in the medium widget.

## Pass 7 — Unresolved Design Decisions

**Rating: 2/10** (a lot is unresolved; that's the point of this pass)

Decisions the design doc doesn't make that will cause problems if deferred to "when we build it."

### Decision 1: Rating scale

**Unresolved.** The plan says "1-5" stars. Is that final? For a personal taste archive, consider 0.5-5 (10 half-star values) — same as RYM's 0.5-5.0 scale. Changing this after you have 100 ratings in the DB is painful. Decide now.

**Concrete options:**
- A) 1-5 integers (simple, common)
- B) 1-10 integers (RYM-adjacent, easy DB storage: Int field, /2 for display)
- C) 0.5-5 with halves (RYM exact, store as Float or Int×2)

Store as an integer (1-10 internally, display as 1-5 or 0.5-5) so you can change the display scale without a migration.

### Decision 2: Promote conditional logic

**Unresolved.** When does the "Add to Best Albums" button appear?
- Always on album detail? (User decides when to promote)
- Only when rating ≥ 4? (App suggests promotion when you rate highly)
- Only when album is 100% listened? (Completion-gated)

The mockup shows a passive prompt ("Add to Best Albums?") with a rating threshold hint. The cleanest UX: always show the button, but change its label/style based on rating. If rating ≥ 4, button is prominent (green). If no rating yet, button is muted with "Rate first?" hint.

### Decision 3: What is "current album"?

**Unresolved.** The home screen shows "the current album." But you might be mid-way through three albums across different playlists. Which one is "current"?

Options:
- Most recently listened to (latest `listened_at` timestamp in `listening_progress`) — simple, correct
- User-pinned — too much friction
- First incomplete album in the current week's playlist — too opinionated

**Use most recently listened to.** It matches the mental model: "what was I just listening to?" The `listened_at` field in `listening_progress` answers this directly.

### Decision 4: Playlist detection on Friday

**Unresolved.** When the user creates a new "New Albums 04/11/26" playlist in Spotify, how does the app know about it? `updateUsersPlaylistDataScheduled` runs on Saturdays at midnight — the user would see an empty app all Friday evening.

Options:
- Pull-to-refresh on the Albums tab (always triggers `updatePlaylistData`)
- Sync on every app open (not just `syncRecentlyPlayed` but also `updatePlaylistData`)
- Both

**Use both: pull-to-refresh + sync on app open.** `updatePlaylistData` is fast (it's just a Spotify playlists API call + Prisma upserts). Triggering it on app open adds ~500ms at most. The Saturday cron then becomes a safety net, not the primary discovery mechanism.

### Decision 5: Dark mode default

**Resolved (by design):** Dark is the correct default for a music app. But add a Settings screen (v1: just a toggle for dark/light + a "Disconnect Spotify" option). Without this, the app feels like a prototype.

### Decision 6: What does "skip to track" mean?

The plan says "skip to track" in album view. This means tapping a track in the list and... what? If using `react-native-spotify-remote`: it can queue the track. If using the fallback (deep link): it opens Spotify to the track via `spotify:track:{id}` URI. Define the target behavior for each SDK path before building the track list UI, otherwise you'll build the wrong tap interaction.

## Required Actions Before Weekend 2 (Expo scaffold)

These must be decided before writing any screen code or you'll build the wrong thing:

1. **Rating scale:** Choose 1-5 integers, 1-10 integers, or 0.5-5 halves. Store as Int×2 internally for flexibility.

2. **NativeWind or plain StyleSheet:** NativeWind recommended. Add it to `package.json` and configure it alongside Expo before touching any UI.

3. **Navigation library:** Expo Router is already committed to (correct). Confirm the tab structure: 4 tabs (Now / Albums / Ratings / Discover), with Discover hidden until Phase 2.

4. **"Current album" definition:** Use `ORDER BY listened_at DESC LIMIT 1` on `listening_progress`. Simple, always correct.

5. **Pull-to-refresh triggers `updatePlaylistData`:** Add this to the Albums tab. Blocks the Friday ritual working correctly.
